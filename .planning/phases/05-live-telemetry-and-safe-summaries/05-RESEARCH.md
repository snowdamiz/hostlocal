## User Constraints (from CONTEXT.md)

### Locked Decisions

### Live event detail
- Live stream follows the currently selected issue in the right sidebar.
- Event detail should be milestone-level (more than stage chips, less than raw command transcript).
- Events should render in a newest-first feed.
- When a run reaches terminal status, keep final live events visible and hand off directly to the final summary.

### Final summary shape
- Final summary should live in the existing issue details panel (right sidebar), not a separate view.
- Summary must include completion outcome, key actions, and validation outcomes.
- Key actions should be concise milestone bullets, not a full raw transcript.
- Validation outcomes should always be explicit, including unavailable/missing states (for example `not-run`/`not-found`).

### Secret-safe visibility
- Sensitive substrings should be masked in-place with a clear marker (for example `[REDACTED]`) instead of dropping entire entries.
- Redaction scope should be strict by default: tokens, API keys, auth headers, cookie/session values, and credential-like environment values.
- UI may support a manual reveal toggle for advanced local debugging workflows.

- Unsafe-content uncertainty policy (when detection confidence is low) was left to implementation discretion.
- Prefer conservative handling during planning because Phase 5 includes SEC-01 guarantees.

## Summary

Phase 4 established a durable runtime state engine and right-sidebar runtime visibility, but current telemetry is limited to stage transitions (`runtime/run-stage-changed`) plus snapshot/history fetches. There is no dedicated live activity feed model, no run-summary contract, and no generalized redaction pipeline for streamed activity text.

Current runtime worker handling in `runtime_boundary.rs` tracks only terminal outcomes from `CommandEvent::Terminated` / `CommandEvent::Error` and ignores richer process output. The bundled sidecar is currently a placeholder script (`hostlocal-worker-aarch64-apple-darwin`) that prints one line and exits, so Phase 5 should plan milestone telemetry generated by runtime lifecycle transitions first, then optionally map structured worker output later.

The safest planning approach is: backend-owned sanitization, backend-owned telemetry persistence, and frontend display-only logic. Persist only sanitized event text. Never send raw lines to the renderer. Compute final summary from sanitized telemetry + canonical terminal state and include explicit validation fallback states now, even before Phase 7 validation implementation.

**Primary recommendation:** Introduce a sanitized telemetry pipeline in Rust (record -> redact -> persist -> emit), then render live feed and terminal summary in `IssueDetailsPanel` using new typed commands/events while preserving keychain-only credential handling.

### Pattern 1: Backend-First Redaction Pipeline
**What:** Sanitize all telemetry text in Rust before persistence and before event emission.
**When to use:** Any runtime event message, sidecar output fragment, or summary action text.
**Implementation guidance:**
- Introduce a `redact_sensitive_text(input) -> { masked, redacted, reasons }` function.
- Match strict categories from locked decisions: token/API key patterns, auth headers, cookie/session values, credential-like env assignments.
- Apply conservative fallback for low-confidence uncertainty: mask the suspicious segment or entire value span.
- Persist only `masked` text, never raw text.

**Example (Rust pseudocode):**
```rust
struct RedactionResult {
    masked: String,
    redacted: bool,
    reasons: Vec<String>,
}

fn redact_sensitive_text(input: &str) -> RedactionResult {
    // 1) Target known secret forms: ghp_/github_pat_/Bearer/Cookie/key=value
    // 2) Mask sensitive substrings with [REDACTED]
    // 3) If uncertain but risky, redact conservatively
}
```

### Pattern 2: Telemetry Event Store + Newest-First Feed
**What:** Persist sanitized milestone events per run with deterministic sequence.
**When to use:** During enqueue/start transitions, workspace prep milestones, sidecar lifecycle milestones, terminal finalization.

**Suggested schema:**
```sql
CREATE TABLE IF NOT EXISTS runtime_run_events (
  event_id INTEGER PRIMARY KEY AUTOINCREMENT,
  run_id INTEGER NOT NULL,
  sequence INTEGER NOT NULL,
  repository_key TEXT NOT NULL,
  issue_number INTEGER NOT NULL,
  stage TEXT NOT NULL,
  kind TEXT NOT NULL, -- stage|worker|validation|system
  message TEXT NOT NULL, -- already redacted
  redacted INTEGER NOT NULL DEFAULT 0,
  redaction_reasons TEXT, -- JSON array or comma string
  include_in_summary INTEGER NOT NULL DEFAULT 1,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP,
  FOREIGN KEY(run_id) REFERENCES runtime_runs(run_id) ON DELETE CASCADE,
  UNIQUE(run_id, sequence)
);

CREATE INDEX IF NOT EXISTS idx_runtime_run_events_issue_created
  ON runtime_run_events(repository_key, issue_number, created_at DESC, event_id DESC);
```

### Pattern 3: Dual Delivery Model (Hydrate + Stream)
**What:** Keep snapshot-like hydration for initial state, then merge live telemetry event payloads.
**When to use:** Repository switch, issue selection, active run updates.

**Contract additions:**
- Command: `runtime_get_issue_run_telemetry({ repositoryFullName, issueNumber, limit })`
- Event: `runtime/run-telemetry` with sanitized payload
- Command: `runtime_get_issue_run_summary({ repositoryFullName, issueNumber, runId? })`

**Why this fits current code:**
- Mirrors existing stage model in `useBoardInteractions` (`hydrate + listen`).
- Preserves existing `runtime/run-stage-changed` contract for Phase 4 behavior.

### Pattern 4: Terminal Handoff to Summary
**What:** On terminal status, retain final telemetry feed and render summary in same details panel.
**When to use:** `terminalStatus != null` for selected run.

**Summary projection rules:**
- `completion`: from `runtime_runs.terminal_status` + `terminal_at`
- `keyActions`: sanitized telemetry milestones with `include_in_summary = 1` (concise bullets)
- `validationOutcomes`: explicit object with fallback states
  - `code`: `pass|fail|timeout|not-found|not-run`
  - `browser`: `pass|fail|timeout|not-found|not-run`
- If no validation telemetry exists yet, return `not-run` explicitly.

### Pattern 5: Security Boundary Preservation
**What:** Keep credential flow unchanged: keyring -> Rust backend only.
**When to use:** Any telemetry/summarization additions.

**Hard rules for planner:**
- Do not add token-bearing fields to frontend event/command payloads.
- Do not include process environment or auth headers in telemetry messages.
- Do not read credentials in frontend; continue using `github_auth.rs` and `github_intake.rs` token resolution path.

### Anti-Patterns to Avoid
- **Frontend-only redaction:** leaks can occur before UI masking.
- **Persist raw then mask on read:** violates SEC-01 data-at-rest expectations.
- **Raw transcript streaming:** conflicts with locked milestone-level requirement and raises leakage risk.
- **Summary as arbitrary worker blob:** hard to sanitize and validate deterministically.
- **Terminal-only telemetry:** fails OBS-01 live observability requirement.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Secret filtering | Ad hoc `contains("token")` checks | Regex-backed redaction rules + conservative fallback | Better coverage for headers/cookies/env-like secrets and lower false negatives. |
| Live feed sync | One-off component-local arrays | Hook-level state + typed commands/events in `useBoardInteractions` | Keeps behavior consistent with existing runtime hydration/listener model. |
| Summary lifecycle | Separate mutable summary state authored by UI | Backend summary projection from canonical run + sanitized events | Avoids drift and keeps security boundary centralized. |
| Event ordering | Client timestamp sorting only | Backend sequence + newest-first query | Deterministic ordering across restart and rapid updates. |

**Key insight:** Phase 5 should treat telemetry as security-sensitive data, not just UI text. Sanitization and shaping must happen before data crosses process boundaries.

## Common Pitfalls

### Pitfall 1: Emitting before masking
**What goes wrong:** Raw text reaches UI/devtools/event subscribers prior to redaction.
**Why it happens:** Redaction is applied only during render or DB read.
**How to avoid:** Redact in Rust before both `insert runtime_run_events` and `app.emit`.
**Warning signs:** Event payloads containing `Bearer`, `ghp_`, `github_pat_`, `Authorization:`.

### Pitfall 2: Feed and summary divergence
**What goes wrong:** Summary bullets do not match visible terminal events.
**Why it happens:** Summary assembled from separate unsynchronized source.
**How to avoid:** Build summary from same sanitized event store + canonical run row.
**Warning signs:** Terminal feed says failed but summary shows incomplete/empty actions.

### Pitfall 3: Non-explicit validation status
**What goes wrong:** Summary omits validation section when no validations ran.
**Why it happens:** Assumes validation phase is not implemented yet.
**How to avoid:** Always return explicit fallback (`not-run` / `not-found`).
**Warning signs:** Null/undefined validation fields in payloads.

### Pitfall 4: Selected-issue drift in sidebar
**What goes wrong:** Live feed shows events from wrong issue/repo.
**Why it happens:** Missing repository/issue filtering in listener merge.
**How to avoid:** Reuse normalized repository filter pattern already used by stage listener and key by `issueNumber`.
**Warning signs:** Feed updates while unrelated issue is selected.

### Pitfall 5: Reveal-toggle scope creep
**What goes wrong:** Manual reveal feature reintroduces raw secret exposure.
**Why it happens:** Revealed text source kept unredacted in store.
**How to avoid:** Phase 5 default should remain redact-only; if reveal is added later, require explicit gated path and never persist plaintext.
**Warning signs:** UI state includes original event text next to masked text.

## Open Questions

1. **Manual reveal toggle policy (from locked "may support")**
   - What we know: strict redaction and SEC-01 are mandatory.
   - What is unclear: whether Phase 5 should ship reveal at all.
   - Recommendation: default to no reveal in Phase 5 plan; if required, gate as explicit opt-in and keep plaintext out of persistence.

2. **Worker telemetry contract maturity**
   - What we know: current sidecar is placeholder script output.
   - What is unclear: whether future worker output will be structured JSON milestones or plain text.
   - Recommendation: phase plan should support milestone events generated by runtime boundary now, with optional parser adapter for future structured worker output.

3. **Summary target run selection**
   - What we know: issue history can contain many runs (newest-first, bounded to 20).
   - What is unclear: whether summary should always follow latest run or selected historical run.
   - Recommendation: default summary to latest run for selected issue; add optional `runId` request parameter for deterministic historical lookup.
