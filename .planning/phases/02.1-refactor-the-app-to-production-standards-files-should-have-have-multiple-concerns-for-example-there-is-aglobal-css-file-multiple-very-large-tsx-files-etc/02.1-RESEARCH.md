## Summary

This phase is primarily a maintainability and architecture hardening pass, not a feature phase. The current codebase has a clear hotspot: `src/components/MainLayout.tsx` is **1,787 lines** and mixes auth polling, repository loading, board derivation, canvas math/rendering, drag/drop orchestration, intake mutations, markdown parsing/highlighting, and full page rendering in one component. Styling is similarly concentrated in `src/App.css` at **1,587 lines** with broad global scope and direct color literals (for example lines 229-240, 270-271, 299-300, 588-600, 751, 772, 778), which conflicts with the project rule to avoid hardcoded colors.

Tailwind is not currently installed/configured in this repo (`package.json` has no `tailwindcss`/`@tailwindcss/vite`). This means Phase 02.1 should explicitly include Tailwind v4 setup and token mapping as part of the refactor, not assume it already exists.

Current automated baseline is stable (`pnpm exec vitest run`: 7 passing tests, `cargo test`: 7 passing tests), so the refactor can be planned as behavior-preserving extraction + styling migration with incremental checkpoints.

**Primary recommendation:** Plan Phase 02.1 as a **strangler refactor** with explicit module boundaries and Tailwind v4 tokenization first, then migrate feature slices one by one (auth/repo/board/canvas/issue panel) behind parity checks.

### Pattern 1: Strangler-fig extraction with feature ownership
**What:** Extract one concern at a time from `MainLayout` into hook/component modules while keeping one integration shell until final cutover.
**When to use:** First pass of Phase 02.1; do not rewrite everything at once.
**Example:**
```ts
// Source: src/components/MainLayout.tsx + Solid createResource docs
export function useRepositories() {
  const [repositories, setRepositories] = createSignal<GithubRepository[]>([]);
  const [selectedRepositoryId, setSelectedRepositoryId] = createSignal<number | null>(null);
  const [error, setError] = createSignal<string | null>(null);

  const loadRepositories = async () => {
    setError(null);
    const all = await githubListRepositories();
    setRepositories(all);
    setSelectedRepositoryId((id) => id ?? all[0]?.id ?? null);
  };

  return { repositories, selectedRepositoryId, setSelectedRepositoryId, error, loadRepositories };
}
```

### Pattern 2: Tailwind v4 token-first styling bridge
**What:** Establish Tailwind v4 + token aliases before large class migration; keep a temporary bridge stylesheet for non-utility edge cases.
**When to use:** Wave 0 / first plan in phase.
**Example:**
```ts

  plugins: [tailwindcss(), solid()],
});
```


### Pattern 3: UI split by render responsibility (container vs presentational)
**What:** Keep side-effect and invoke logic in hooks/containers; move JSX-heavy views into presentational components with typed props.
**When to use:** Board, sidebar, and issue panel extraction.
**Example:**
```tsx
interface RepositoryListProps {
  repositories: GithubRepository[];
  selectedRepositoryId: number | null;
  onSelect: (id: number) => void;
}

export function RepositoryList(props: RepositoryListProps) {
  return (
    <For each={props.repositories}>
      {(repo) => (
        <button onClick={() => props.onSelect(repo.id)}>{repo.name}</button>
      )}
    </For>
  );
}
```

### Anti-Patterns to Avoid
- **Big-bang file replacement:** too risky for Phase 2 drag/intake behavior.
- **Mixing migration and feature work:** refactor scope should not absorb Phase 3+ runtime logic.
- **Reintroducing global styling sprawl in new files:** new Tailwind classes must still resolve through shared tokens.
- **Leaving parsing/highlighting logic embedded in view layer:** makes testing and security hardening difficult.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Tailwind compiler wiring | Custom PostCSS/Vite plugin pipeline guessing | Official `@tailwindcss/vite` setup | Officially supported path for Tailwind v4 + Vite. |
| Theme token plumbing | Ad-hoc color constants in each component | Tailwind `@theme` + existing CSS variables | Keeps one token source and enforces no-hardcoded-color rule. |
| Complexity/file-size policing | Custom scripts for function/line counting | ESLint `complexity` + `max-lines` rules | Standard, maintainable quality gates. |
| Async state orchestration in giant component | One mega component with all side effects | Solid hook modules + `createResource`/signals by feature | Reduces coupling and makes behavior testable per concern. |

**Key insight:** Production quality here means predictable change boundaries. Splitting code without enforcing style/token and complexity guardrails will regress quickly.

## Common Pitfalls

### Pitfall 1: Breaking intake trigger semantics while extracting drag/drop
**What goes wrong:** `todo -> inProgress` gating or optimistic rollback behavior changes.
**Why it happens:** Drop logic gets split without keeping exact decision order.
**How to avoid:** Preserve `performColumnDrop` contract and add focused tests before/after extraction.
**Warning signs:** Cards remain in wrong column or rejection toasts stop appearing.

### Pitfall 2: Tailwind migration bypasses token rules
**What goes wrong:** New utility classes use literal arbitrary colors (`bg-[#...]`).
**Why it happens:** Fast migration without token mapping layer.
**How to avoid:** Define token aliases in `@theme` first; block arbitrary color literals in review/lint.
**Warning signs:** Hardcoded hex strings reappear in TSX/CSS.

### Pitfall 3: Canvas/drag interactions regress on mobile or pointer edge cases
**What goes wrong:** Pan/zoom/drag ghost behavior becomes jittery or stuck.
**Why it happens:** Event listeners and pointer capture cleanup are moved inconsistently.
**How to avoid:** Extract canvas and drag hooks with explicit mount/cleanup parity and interaction smoke tests.
**Warning signs:** `is-card-dragging` class sticks; pointer-up no longer ends drag.

### Pitfall 4: Partial CSS migration leaves duplicate competing styles
**What goes wrong:** Same UI receives both old class selectors and new Tailwind utilities.
**Why it happens:** No migration boundary or “done” criteria per slice.
**How to avoid:** Migrate by vertical slice and delete legacy selectors immediately for that slice.
**Warning signs:** Style flicker, specificity fights, hard-to-predict hover/focus states.

### Pitfall 5: Refactor phase drifts into unrelated security/runtime initiatives
**What goes wrong:** Phase scope balloons and stalls delivery.
**Why it happens:** “Production standards” interpreted too broadly.
**How to avoid:** Keep this phase to structure/style/testability guardrails; track non-refactor concerns as explicit follow-up tasks.
**Warning signs:** Plans start adding Phase 3+ behaviors or backend feature changes.
